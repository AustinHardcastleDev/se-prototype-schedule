### Learnings:
**touch-none CSS Class Blocks All Touch Interactions:** The Tailwind `touch-none` class sets `touch-action: none`, which disables ALL default touch behaviors including:
- Scrolling
- Pinch-to-zoom
- Double-tap-to-zoom
- Browser-managed pan/swipe gestures

This was originally added to prevent scroll interference during drag operations, but it blocked legitimate scroll on empty areas.

**Pointer Event Handlers Don't Block Scroll by Default:** The presence of `onPointerDown/Up/Leave` handlers does NOT inherently block scrolling. The browser can simultaneously:
1. Track pointer events for app logic (long-press detection)
2. Process scroll gestures natively

The key is NOT applying `touch-none` or `touch-action: none`. With `touch-action: auto` (default), the browser handles both.

**Long-Press Pattern Compatible with Scroll:** The long-press implementation using setTimeout and pointer handlers works correctly for detecting intentional holds without blocking scroll:
```javascript
handlePointerDown → start 500ms timer
handlePointerUp → cancel timer (if released before 500ms)
handlePointerLeave → cancel timer (if finger moves during scroll)
```

When scrolling, the finger moves across the surface, triggering `pointerleave` which cancels the timer. Only a stationary hold for 500ms triggers the long-press callback.

**Testing Touch Interactions with Playwright:** Automated browser testing has limitations for touch gestures:
- `page.mouse.down()` + `setTimeout` doesn't perfectly simulate touch holds
- `page.touchscreen` requires explicit hasTouch context setup
- Programmatic testing (CSS checks, scroll property manipulation) is more reliable for verifying fixes

For touch-specific UX, combine automated property checks with manual device testing.

**Minimal Fix Principle:** The fix required changing only ONE word (removing `touch-none`). No refactoring of pointer event handlers was needed. When debugging touch issues:
1. Check for `touch-action: none` or `touch-none` class first
2. Verify pointer-events CSS isn't set to 'none'
3. Only refactor handlers if CSS isn't the issue

**Custom Dropdown Component Pattern:** Built a reusable dropdown that encapsulates:

- Outside click detection with useEffect + refs

3. Consistent behavior across browsers and platforms
4. Better accessibility control

**Outside Click Handler Pattern:**
```javascript
useEffect(() => {
  const handleClickOutside = (event) => {
    if (dropdownRef.current && !dropdownRef.current.contains(event.target)) {
      setIsOpen(false)
    }
  }
  if (isOpen) {
    document.addEventListener('mousedown', handleClickOutside)
  }
  return () => {
    document.removeEventListener('mousedown', handleClickOutside)
  }
}, [isOpen])
```

Only attach the listener when dropdown is open to avoid unnecessary event handling. Clean up on unmount or when closed.

**Prop Function Pattern for Dynamic Values:** The `getColorForOption` prop allows the parent to provide a function that maps option values to colors:
```javascript
getColorForOption={(key) => {
  const et = eventTypes.find(t => t.key === key)
  return et?.borderColor
}}
```

This keeps the CustomDropdown generic while supporting specialized use cases (event type colors) without hardcoding domain logic in the component.

**ARIA for Custom Controls:** Custom dropdowns need proper ARIA attributes:
- `role="listbox"` on the dropdown menu
- `role="option"` on each item
- `aria-expanded` on the trigger button
- `aria-selected` on the currently selected option
- `aria-haspopup="listbox"` on the trigger

These attributes ensure screen readers announce the dropdown state and allow keyboard-only users to navigate effectively.

**Transition Indicators:** The down arrow rotates 180° when dropdown opens:
```jsx
<svg className={`transition-transform ${isOpen ? 'rotate-180' : ''}`}>
```

This visual feedback helps users understand the dropdown state without needing to read text.

**Scrollable Dropdown with max-height:** Using `max-h-60 overflow-y-auto` ensures long lists (like team members) don't overflow the viewport. The menu becomes scrollable while keeping a reasonable height.

### Learnings:

**Horizontal Scroll with Fixed Columns Pattern:**
The key to horizontal scroll with a fixed column (time labels) is separating the layout into:
1. Fixed column outside the scroll container
2. Scrollable container with overflow-x-auto
3. Inner content with explicit minimum width

This differs from vertical sticky headers which use `position: sticky`. For horizontal, you need actual DOM separation.

**Bidirectional Scroll Synchronization:**
When synchronizing scroll between two containers:
```javascript
const sync = (source, target) => {
  if (target.scrollLeft !== source.scrollLeft) {
    target.scrollLeft = source.scrollLeft
  }
}
```

The conditional check `if (target.scrollLeft !== source.scrollLeft)` is CRITICAL. Without it, setting scrollLeft triggers another scroll event, creating an infinite loop. The check breaks the loop because once they match, no assignment occurs.

**flex-shrink-0 for Fixed-Width Columns:**
When using flexbox with `overflow-x-auto`, child elements will shrink by default even with `minWidth`. You MUST add `flex-shrink-0` to prevent columns from shrinking below their minimum width:
```jsx

```

Without `flex-shrink-0`, the columns would shrink to fit the viewport and never trigger horizontal scroll.

**scrollbar-hide Utility Pattern:**

```css
.scrollbar-hide::-webkit-scrollbar { display: none; }

This creates a cleaner UI where scrolling works but scrollbars are invisible. Users can still scroll with trackpad, mouse wheel, or touch gestures.

  if (isOpen) {
    document.addEventListener('mousedown', handleClickOutside)
  }
  
  return () => {
    document.removeEventListener('mousedown', handleClickOutside)
  }
}, [isOpen])
```

Key points:
- Only attach listener when popup is open (performance)
- Use `mousedown` instead of `click` for better UX (closes before click completes)
- Check `contains()` to determine if click was inside
- Clean up listener on unmount or when popup closes

**Distinguishing Click from Drag in @dnd-kit:**
The @dnd-kit library captures all pointer events for draggable elements, which can interfere with click handlers. To reliably distinguish clicks from drags:

```javascript
const pointerDownRef = useRef(null)
const hasDraggedRef = useRef(false)

const dragListeners = {
  ...listeners,
  onPointerDown: (e) => {
    pointerDownRef.current = { x: e.clientX, y: e.clientY }
    hasDraggedRef.current = false
    if (listeners.onPointerDown) listeners.onPointerDown(e)
  },
  onPointerMove: (e) => {
    if (pointerDownRef.current) {
      const dx = Math.abs(e.clientX - pointerDownRef.current.x)
      const dy = Math.abs(e.clientY - pointerDownRef.current.y)
      if (dx > 5 || dy > 5) {
        hasDraggedRef.current = true
      }
    }
    if (listeners.onPointerMove) listeners.onPointerMove(e)
  }
}

const handleClick = (e) => {
  if (!isDragging && !hasDraggedRef.current) {
    onEventClick(event)
  }
  hasDraggedRef.current = false
  pointerDownRef.current = null
}
```

**Key Points:**
- **5px threshold:** Prevents accidental drags from blocking clicks due to micro-movements
- **Reset in onClick:** Ensures clean state for next interaction
- **Call original listeners:** Maintains @dnd-kit functionality by calling `listeners.onPointerMove(e)`
- **useRef over useState:** Avoids re-renders for internal tracking state

**Why e.detail === 1 Check Alone Is Insufficient:**
The existing check `if (!isDragging && e.detail === 1)` only prevents clicks during active drag (`isDragging = true`). However, it doesn't prevent clicks AFTER a drag completes. The sequence is:
1. User drags event (isDragging becomes true)
2. User releases (isDragging becomes false, onClick fires)
3. Without hasDraggedRef check, onClick would trigger after every drag

The hasDraggedRef prevents this by tracking whether the current interaction involved movement.

**Integration with Existing Drag Listeners:**
The fix integrates with @dnd-kit listeners by:
1. Spreading original listeners: `...listeners`
2. Wrapping specific listeners (onPointerDown, onPointerMove) to add tracking
3. Calling original listener functions: `if (listeners.onPointerMove) listeners.onPointerMove(e)`

This ensures drag functionality continues working while adding click detection.

**Alternative Approaches Considered:**
1. **PointerMove distance in @dnd-kit config:** @dnd-kit has `activationConstraint.distance` but this only affects when drag STARTS, not click detection
2. **Separate click overlay:** Could add a transparent overlay for clicks, but adds complexity and z-index issues
3. **onClickCapture:** Doesn't help because drag listeners run before capture phase

The chosen approach (movement tracking) is the most reliable and least invasive.
